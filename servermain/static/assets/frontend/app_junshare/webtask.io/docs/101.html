Webtasks 101

Execute custom Node.js code with HTTP POST

For customized documentation and ready to run samples, please log in.

Overview

You can get an overview of the webtask concepts and design on the Rethinking backend with webtasks blog post. Read the Sandboxing code in the era of containers blog post on AWS Startup Spotlight blog to get an insight into how webtasks are implemented.

Hello, world

You can use your webtask token to run custom Node.js backend code and get back the result as JSON. Submit your webtask code for execution using HTTPS POST as follows:

curl https://webtask.it.auth0.com/api/run/{webtask_container} -H "Authorization: Bearer {webtask_token}" --data-binary 'return function (cb) { cb(null, "Hello, world!"); }'
Copy

The Node.js code

The Node.js code submitted for execution in the body of the HTTP POST request must return a JavaScript function. The function accepts a callback parameter. Your code must call the callback with an optional error and a result. The result is serialized to JSON and returned in the body of the HTTP POST response:

return function (cb) {
  cb(null, { any: 'thing that can', be: 'serialized to JSON' });
}

Security and isolation

Your code executes in an environment isolated from other callers. The first segment of the URL path of the HTTPS post request ( in the example above) identifies the webtask container name, which defines the isolation boundary. No two HTTP POST requests using different webtask container names will be allowed to share in-memory or on-disk data. They will also be guaranteed their fair share of computing resources. Multiple requests using the same webtask container name may or may not execute in the same environment. This means you can optimistically use in-memory caching across requests using the same webtask container name.
